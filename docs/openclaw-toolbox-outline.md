# OpenClaw 工具箱大綱：我給 OpenClaw 的能力

對應既有文章：[讓 AI 當我的工程實習生：在公司導入 OpenClaw（龍蝦）的實戰心得](https://it-monk.com/ai-engineer-intern-openclaw/)

本篇大綱用於專訪或寫文時，對齊「已有文章寫了什麼」與「下一篇可延伸什麼」。  
✓ = 既有文章已做／已寫　→ = 可作為下一篇延伸或補強

---

## 1. OpenClaw 的工具架構

AI 透過工具操作系統（Jira、Git、Jenkins、Telegram），而不是直接改程式或自己拍板。既有文章：餵專案架構知識、給 Git 權限、用 Telegram 當操作台、用 Jenkins 當既有部署馬達。

→ **下一篇可寫**：明確畫出「工具層 vs 模型」的架構，或一兩段說明「為什麼用工具、不讓 AI 直接動系統」。

---

## 2. Repo／專案結構與知識

既有文章：讓龍蝦「看得懂專案」——各 repo 角色、目錄結構、關鍵模組；用 **knowledge** 載入架構說明與約定，減少長對話與 token。

→ 若有做或想寫「Code Search」工具（搜 repo、搜程式碼），可獨立成一節；沒有就維持「專案結構＋knowledge」即可。

---

## 3. Patch／Code Edit（程式碼修改方式）

既有文章只寫「修改程式碼、開 branch、發 PR」，沒寫是否用 diff/patch 或具體編輯介面。

→ 別人建議的「用 diff/patch 控制 AI 的程式碼修改」可當作：你**實際怎麼做**（例如透過哪種介面／工具）或**未來想怎麼做**的延伸題。

---

## 4. Git Workflow 工具

既有文章已寫：開 branch → 改 code → 跑測試 → 發 PR → 人 review → 通過後你在 Telegram 指示部署、龍蝦觸發 Jenkins。

→ 可標明為「你給 OpenClaw 的 Git 工作流」，必要時補一兩句實務（branch 命名、PR 範本等）。

---

## 5. Test Runner 工具

既有文章：龍蝦會執行單元測試／整合測試，改完自己驗證。

→ 可標明為「你給 OpenClaw 的能力之一」：改完程式後能自己跑測試，失敗就不該進到 PR/部署。

---

## 6. 文件與知識搜尋

既有文章：用 knowledge 載入架構與約定、讀規格書、**Jira 描述＋留言**當任務介面（修復方向、模組提示寫在留言）。

→ 可收斂為「你給 OpenClaw 的文件與知識能力」= 專案 knowledge + 規格書 + Jira 留言；若有「搜專案文件／規範」的工具再補一節。

---

## 7. MCP：連接公司系統

既有文章已做：Jira MCP（讀單、開單、更新狀態與留言）、Jenkins（觸發 pipeline）、Telegram（下指令、看進度）。

→ 可明確列成「我給 OpenClaw 的 MCP 能力」：Jira、Jenkins、Telegram；若有 PR 查詢或內部 API 再列上。

---

## 8. Guardrails（安全與邊界）

既有文章有做但沒用這個詞：**人工審核 gate**（PR 人 review、部署由人決定）、任務邊界（拆 Jira、拆 session）、**硬性規定**（例如讀 Jira 一定要看留言）、token/記憶當系統設計（knowledge、修復計劃書、一任務一 session）。文末也提到資安與理解錯誤的隱憂、只讓它做可承擔後果的任務。

→ 可收斂為「你給 OpenClaw 的 Guardrails」= 權限邊界（不讓 AI 拍板部署）+ 任務邊界（拆單、拆 session）+ 介面約定（描述＋留言、計劃書）+ 對資安與錯誤的事前設限。

---

## 專訪／寫文時可用的方向

- **已有文章寫過的**：專訪可問「當時為什麼這樣設計」「現在回頭看會想加強哪一塊」。
- **別人建議但你還沒做的**（例如明確的 Code Search、diff/patch 流程）：專訪可問「這些你有打算做嗎？卡在哪？」或「你希望讀者先知道哪幾項能力就好？」。
- **標題與主軸**：「我給 OpenClaw 的能力」可扣在這八項；一篇講「工具箱長怎樣、我給了哪些能力」，與現有「實戰流程、坑與建議」那篇互補。
