## 來自「AI 能完全取代程式設計師嗎」專訪（約 2026-03）

- **如何把任務與規格講給 AI 聽**  
  工程師怎麼把需求、脈絡說清楚，讓 AI 能正確理解並執行；避免「修好」這類模糊指令導致 AI 卡在單一專案或偏離。

- **工程師怎麼訓練／指揮 AI 補齊規格**  
  第一階段規格缺漏時，如何指揮 AI 補齊、而不是放任 AI 自己猜；監督產品不失控、不偏離的實務做法。

## 來自「工程師的影響力如何擴大」專訪（約 2026-03）

- **「5 分鐘打造 app」背後的風險**  
  可維護性、長期維運成本、app 能否拓展；社群敘事未體現的隱藏問題。

- **對積極度低的成員：例行優化 + 問期待與改進**  
  配置例行工作的優化與修改，持續問對現況的期待與流程改進方針，讓其感受對團隊的責任感。

## 來自「OpenClaw 工具箱：我給 OpenClaw 的能力」專訪（約 2026-03）

- **讓 AI 自己寫 knowledge、人再審**  
  請龍蝦看專案 commit 歷史，歸納修復推理原則並提交到 knowledge repo；人讀完摘要錯誤、留言糾正。高效率且有利複製下一隻龍蝦。

- **訓練新人的架構文件給龍蝦用**  
  公司 git／Jira／規格文件位置等產品架構關聯文件，改寫後作為龍蝦必讀 knowledge。

- **為龍蝦製作工具與腳本**  
  部署除 Jenkins API 外再做本地 shell script 減 token；專案性質用 .json（名稱、描述、用途、git、tag）方便運用。完全不為它製作工具其實是很高的成本。

- **與龍蝦一起維護 skill 專案、繁殖第二隻 OpenClaw**  
  請龍蝦把用到的工具（不含敏感資訊）與 skill 寫進獨立 git repo；雖尚未實作，但很有機會以此繁殖第二隻 OpenClaw。
